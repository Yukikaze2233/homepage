# Homepage 架构说明

## 项目定位
这是一个基于原生 `HTML + CSS + JavaScript` 构建的静态个人主页项目。整个站点不依赖前端框架，也没有构建流程，核心特点是结构简单、资源清晰、易于直接修改。

---

## 整体架构

### 1. 页面入口层
- `index.html` 是整个网站的唯一入口文件。
- 页面内容主要分为两大区域：
  - 左侧信息区：头像、地点、时间线
  - 右侧主内容区：欢迎语、描述、社交入口、项目卡片、技能展示
- 文案、链接、卡片结构等核心内容都直接维护在 `index.html` 中。

### 2. 样式层
- `static/css/style.css`
  - 负责页面布局
  - 卡片样式
  - 动画效果
  - 响应式适配
  - 移动端显示规则

- `static/css/root.css`
  - 负责主题变量
  - 背景图配置
  - 渐变色配置
  - 卡片透明度与全局遮罩透明度

这种拆分方式把“结构样式”和“主题参数”分开，后续调整视觉风格时更容易维护。

### 3. 交互层
- `static/js/script.js` 负责：
  - 明暗主题切换
  - 图片弹窗预览
  - 卡片按压反馈
  - 页面加载过渡
  - 控制台欢迎信息输出

交互逻辑较轻，主要用于增强体验，不参与页面内容生成。

### 4. 资源层
- `static/img/`
  - 存放头像、背景图、项目卡片图片、网站图标、弹窗图片等位图资源

- `static/svg/`
  - 存放 SVG 资源，例如技能图、装饰图等

- `static/fonts/`
  - 存放本地字体文件，用于标题和页面文字渲染

### 5. 部署层
- `Dockerfile`
  - 将整个静态站点打包进 Caddy 容器镜像

- `docker-compose.yaml`
  - 用于启动 Caddy 服务容器

- `Caddyfile`
  - 定义域名、静态资源根目录以及文件服务规则

这意味着项目既可以走纯静态托管，也可以走基于 Caddy 的容器部署。

### 6. 贪吃蛇动画层
- 页面中的贪吃蛇动画资源来自：
  - `static/svg/snake-Light.svg`
  - `static/svg/snake-Dark.svg`
- 这两个文件不是运行时动态计算出来的，而是**预生成的 SVG 动画文件**。
- 当前文件内部可以直接看到：
  - `Generated with https://github.com/Platane/snk`
  - 内嵌的 `<style>`
  - 大量 `@keyframes`

这说明当前项目中的贪吃蛇动画本质上是一个“生成好的成品 SVG”，网站运行时只负责显示它，而不负责重新计算动画路径。

#### 动画生成逻辑
Platane/snk 的生成过程可以理解为：
1. 读取 GitHub contribution graph 数据
2. 将 contribution heatmap 当作一个网格棋盘
3. 求解一条蛇的移动路径，让蛇按顺序“吃掉”亮起的格子
4. 为每个格子生成时间轴
5. 把时间轴展开成 SVG 内部的 `@keyframes`
6. 输出最终的 `snake.svg`

#### 为什么本地 SVG 会自己动
因为动画规则已经直接写进 SVG 文件里：
- 每个格子都有对应的 class
- SVG 内部带有 `@keyframes`
- 浏览器加载 SVG 时，SVG 内部样式就会自动生效

所以页面中只需要像普通图片一样引用：

```html
<img id="tanChiShe" src="./static/svg/snake-Light.svg" alt="">
```

主题切换时，再通过 `script.js` 把它切换到暗色版本：
- `snake-Light.svg`
- `snake-Dark.svg`

这也是为什么这个动画被称为“预生成”：
- 网站运行时不需要重新请求 GitHub contribution graph
- 不需要再求解蛇路径
- 不需要额外动画脚本参与计算
- 浏览器只负责渲染已经生成好的 SVG 动画文件

## 外部资源说明
- 图标资源主要使用了 **iconfont** 风格的 SVG 图标

- 贪吃蛇贡献图动画资源基于 **Platane/snk** 生成的 SVG 成品文件
